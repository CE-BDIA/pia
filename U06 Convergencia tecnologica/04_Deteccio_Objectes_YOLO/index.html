    04 Deteccio Objectes YOLO

    1. Introducción. Descripción breve de YOLO

    https://docs.ultralytics.com/es

    YOLO (You Only Look Once) es una familia de modelos de deep learning enfocados en la detección de objetos en imágenes o videos. Se caracteriza por su velocidad y precisión en tiempo real, ya que el modelo procesa toda la imagen de una sola vez para predecir las regiones donde se encuentran los objetos y su clase.

    Cómo se entrena un modelo YOLO

    • Recopilación de datos: Se juntan imágenes etiquetadas con las coordenadas de los objetos y sus categorías.
    • Preparación de datos: Se estructuran las etiquetas en un formato compatible (por ejemplo, COCO o YOLO).
    • Configuración del entrenamiento: Se elige la arquitectura YOLO (p. ej. YOLOv5, YOLOv7, YOLOv8, etc.), el tamaño del lote (batch size) y los hiperparámetros (épocas, tasa de aprendizaje, etc.).
    • Proceso de entrenamiento: Con cada batch de imágenes, la red ajusta los pesos para minimizar la diferencia entre las predicciones y las etiquetas reales.
    • Validación y ajuste: Se evalúa el rendimiento en un conjunto de validación para ajustar hiperparámetros y evitar sobreajuste.

    2.1. Formato txt

    El formato TXT que emplea YOLO (tanto v5, v7, v8, etc.) para las etiquetas de detección de objetos sigue una estructura muy concreta. Cada imagen tiene asociado un fichero .txt con el mismo nombre (salvo la extensión). Dentro del fichero de etiquetas, cada línea describe un bounding box (caja delimitadora) de un objeto detectado/anotado en la imagen. La línea se compone de:

    Bash
    1
    class_id  x_center  y_center  width  height
    todos los valores están normalizados en el rango 0-1 con respecto a las dimensiones de la imagen:

    • class_id: Identificador de la clase (objeto) en formato entero (0, 1, 2...).
    • Si solo tienes una clase (p. ej. “license_plate”), será siempre 0. Si por ejemplo tienes matriculas y faro_izq y faro_der, matricula seria el 0 y faro_izq seria el 1 y el faro_der el 2
    • x_center: Coordenada X del centro de la caja, normalizada (dividida entre el ancho de la imagen).
    • y_center: Coordenada Y del centro de la caja, normalizada (dividida entre la altura de la imagen).
    • width: Ancho de la caja, también normalizado (dividido entre el ancho de la imagen).
    • height: Alto de la caja, normalizado (dividido entre la altura de la imagen).

    Exemple d'una BB

    Imagina que tu imagen mide 1280 px de ancho por 720 px de alto, y la caja delimitadora que te interesa va de (x1=600, y1=200) a (x2=750, y2=300).

    Valores reales:

    • Ancho de la caja = x2 - x1 = 150 px
    • Alto de la caja = y2 - y1 = 100 px
    • Centro X = (x1 + x2)/2 = (600 + 750)/2 = 675 px
    • Centro Y = (y1 + y2)/2 = (200 + 300)/2 = 250 px

    Valores normalizados: - Ancho normalizado = 150 / 1280 ≈ 0.1172 - Alto normalizado = 100 / 720 ≈ 0.1389 - Centro X normalizado = 675 / 1280 ≈ 0.5273 - Centro Y normalizado = 250 / 720 ≈ 0.3472

    Si class_id=0, la línea en tu archivo .txt quedaría:

    Text Only
    1
    0 0.5273 0.3472 0.1172 0.1389

    En definitiva, cuando YOLO entrena (o infiere), buscará estos archivos .txt para saber dónde están los objetos (en tu caso, las matrículas) y con qué clase asociarlos. Esto es fundamental para que el modelo aprenda de forma supervisada.

    Luego, en inferencia o prediccion, el modelo generará sus propias coordenadas de salida, también normalmente en un sistema similar, pero te las proveerá en escala absoluta o normalizada según la librería que estés usando.

    2.2. data.yaml

    En el caso de YOLO (especialmente en variantes como YOLOv5 o YOLOv8), el archivo .yaml que se encuentra en la carpeta principal (o en la raíz del dataset) sirve para describir la configuración del conjunto de datos.

    Por lo general, este archivo .yaml contiene información como:

    • Rutas de los datos:
    • La ruta o path donde se ubican las imágenes de entrenamiento (train).
    • La ruta donde se ubican las imágenes de validación (val).
    • (Opcionalmente) la ruta a las imágenes de prueba (test).
    • Cantidad y nombre de las clases:
      • Un listado con los nombres de cada clase (por ejemplo, ['perro', 'gato', 'persona']).
      • El número total de clases en el dataset.
    • Parámetros adicionales (opcional):
      • Rutas de anotaciones, si se manejan en diferentes carpetas.
      • Configuraciones específicas para el entrenamiento (aunque este tipo de info a veces va en otro archivo distinto).
      • Identificadores o campos extras que te permitan integrar el dataset en un flujo de trabajo más complejo.

    2.3. Augmentació del Dataset

    Como sabemos, una de las partes mas complicadas es la generación del dataset. Podemos incrementar la cantidad de imagenes de dos modos

    2.3.1. Aumentación en línea (on-the-fly)

    Este tipo de aumentación se hace durante el entrenamiento. La mayoría de implementaciones recientes de YOLO (p. ej., YOLOv5, YOLOv7, YOLOv8) ya incluyen una serie de transformaciones aplicadas automáticamente.

    Configuración de hiperparámetros

    En muchos repositorios (como el de YOLOv5), encuentras un archivo de hiperparámetros, a menudo llamado hyp.yaml o similar. Dentro de este archivo, puedes ajustar distintos parámetros de aumentación como:

    • hsv_h, hsv_s, hsv_v: Aumentos de tono, saturación y valor en espacio de color HSV.
    • degrees: Rotación.
    • translate: Traslación (movimiento de la imagen).
    • scale: Escalado.
    • shear: Efecto de cizalladura (shear).
    • flipud: Volteo vertical.
    • fliplr: Volteo horizontal.

    Por ejemplo, en YOLOv5 (dentro de su repo oficial) existe un archivo hyp.yaml con parámetros predefinidos. Puedes editarlo para ajustar la intensidad de cada transformación.

    YAML
     1
 2
 3

 5

    # Ejemplo (simplificado) hyp.yaml
augments:
  degrees: 0.0
  translate: 0.1
  scale: 0.9
  shear: 0.0
  flipud: 0.0
  fliplr: 0.5
  hsv_h: 0.015
  hsv_s: 0.7
  hsv_v: 0.4

    Al entrenar tu modelo (por ejemplo python train.py --hyp hyp.yaml ...), YOLO aplicará automáticamente estas transformaciones al cargar cada lote (batch) de imágenes.

    2.3.2. Aumentación previa (offline)

    La idea aquí es crear copias aumentadas de las imágenes (y sus etiquetas correspondientes) antes de iniciar el entrenamiento. Para ello, se pueden usar librerías como Albumentations, imgaug o las transformaciones de torchvision.

    Flujo general con Albumentations Instalación:

    Bash
    1
    pip install albumentations

    Configurar un script de aumento:

    • Definir las transformaciones que deseas aplicar (p. ej., rotaciones, flips, cambios de brillo, etc.).
    • Cargar cada imagen y su anotación, aplicar la transformación y luego exportar la imagen aumentada y su nueva anotación. Tras cada transformación, es probable que las cajas delimitadoras (bounding boxes) cambien, por lo cual debes guardarlas con el nuevo formato (por ejemplo, en un archivo .txt con formato YOLO).
    • Guardar imágenes aumentadas: Puedes almacenarlas en carpetas como images/train_aug, por ejemplo, y usar esta carpeta ampliada para entrenar.

    Este método crea nuevos archivos en disco, aumentando explícitamente el número de imágenes en tu dataset. Sin embargo, ten en cuenta que esto ocupa más espacio y lleva tiempo adicional para la generación. A cambio, puede ser beneficioso si quieres realizar un control muy preciso de tus datos o si entrenas con plataformas que no admiten fácilmente la aumentación en línea.

    Para etiquetar un conjunto de imágenes, podemos utilizar herramientas como:

    • https://github.com/HumanSignal/labelImg
    • https://www.makesense.ai/
